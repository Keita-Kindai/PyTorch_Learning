作成日: 2026-06-28  
目的: Lesson単位で残っているPyTorch学習記録を、就職活動で説明しやすい「学習成果・小規模MLプロジェクト」として見せられる形に整理するための提案です。

この提案段階では、既存のPythonコード・データ・モデルファイルは変更しません。

## 結論

このフォルダは、単に「Lessonごとに学習しました」と見せるよりも、次のように再構成した方が強く見えます。

1. メイン成果物: FoodVision Mini
   - pizza / steak / sushi の3クラス画像分類

   - Gradioデモあり
   - Hugging Face Spacesに載せられる構成あり
   - 学習済み `.pth` あり

2. 学習過程の証拠: PyTorch基礎から画像分類まで
   - MNIST分類
   - CNNの自作
   - カスタムDataset実装
   - DataLoader、transform、train/test loop
   - TensorBoardによる実験ログ

3. 補助成果物: 復習プロジェクト
   - `Lesson9_recap` に、学習後に自力で再構成しようとした跡がある
   - コメント量が多く、理解しながら実装した証拠として使いやすい
   - ただし現状はモデル出力クラス数や評価処理など要確認点があるため、メイン成果物よりは「復習・検証ログ」として扱うのが安全

## 整理方針

### 方針A: 元フォルダを残して、就活用ビューだけ作る

既存のLessonフォルダはそのまま残し、新しく `portfolio/` だけ作る方法です。

推奨構成:

```text
portfolio/
  foodvision-mini/
    README.md
    app/
      app.py
      model.py
      requirements.txt
    artifacts/
      model-card.md
      metrics.md
    notes/
      learning-log.md
      dataset.md
```

メリット:

- 既存コードを壊しにくい
- 就活で見せる導線だけ整えられる
- Lesson形式の学習履歴も残せる

デメリット:

- 同じようなファイルが複数残るため、整理後も完全にスリムにはならない

現時点ではこの方針が最も安全です。

### 方針B: Lessonを再分類して、学習ロードマップとして見せる

Lessonごとのまま残しつつ、ルートREADMEで次のように整理します。

```text
01_pytorch_basics/
02_mnist_cnn/
03_custom_dataset/
04_transfer_learning/
05_experiment_tracking/
06_deployment_gradio/
archive_lessons/
```

メリット:

- 学習の積み上げが伝わる
- コメントが多いファイルを「理解の記録」として活かせる

デメリット:

- ファイル移動が多くなる
- importパスが壊れる可能性がある
- 実行可能性の検証が必要

### 方針C: 成果物だけを別リポジトリに切り出す

FoodVision Miniだけを新しいリポジトリに切り出す方法です。

推奨構成:

```text
foodvision-mini/
  README.md
  app.py
  model.py
  requirements.txt
  examples/
  models/
  docs/
    dataset.md
    training.md
    evaluation.md
```

メリット:

- 採用担当者に見せやすい
- READMEが短く、成果が伝わりやすい
- Hugging Face SpacesやGitHubに載せやすい

デメリット:

- 学習過程のコメントや試行錯誤は別途リンクする必要がある
- モデルファイルが大きい場合、Git LFSなどの検討が必要

## 推奨する実行順

### Step 1: 棚卸しREADMEを追加

まずルートに `README.md` を作り、以下を明記します。

- このフォルダの目的
- メイン成果物はFoodVision Miniであること
- Lessonごとの意味
- どのファイルを見ればよいか
- 大きいデータやログは学習記録として残していること

この段階ではコード移動なし。

### Step 2: FoodVision MiniのREADMEを整える

`Lesson8/demos/my_project/README.md` または新規 `portfolio/foodvision-mini/README.md` に、以下をまとめます。

- 概要
- 使用技術
- データセット
- モデル構成
- 学習方法
- 評価指標
- 推論方法
- デモURL
- 今後の改善点

### Step 3: 評価指標を確認する

現状、TensorBoardログと学習コードは残っていますが、就活資料に書くには評価値を再確認した方がよいです。

確認したい項目:

- test accuracy
- test loss
- 推論時間
- どのモデルが最終モデルか
- どの `.pth` が提出・公開用か

評価値が不明なままの場合は、READMEには `要確認` と書くべきです。

### Step 4: データセット出典・ライセンスを確認

就活やGitHub公開で重要です。

確認したい項目:

- pizza / steak / sushi データの出典
- Kaggle由来か、チュートリアル由来か
- 再配布してよいか
- GitHubに画像を置いてよいか

画像データの再配布が不明な場合、公開リポジトリにはデータ本体を置かず、取得手順だけを書く方が安全です。

### Step 5: 不要物を分離する

公開前には以下を整理候補にします。

- `__pycache__/`
- `.DS_Store`
- TensorBoardの古いruns
- 重複したデータセット
- 重複したGradioデモ
- 外部教材リポジトリ `Lesson6/pytorch-deep-learning`

ただし、削除は最後でよいです。まずは `archive/` に寄せる方が安全です。
